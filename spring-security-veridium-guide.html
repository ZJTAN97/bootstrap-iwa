1 Architecture Overview

Spring Security sits entirely in the Servlet Filter chain — every HTTP request passes through a series of security filters before reaching your controller. The filters run in a defined order and each can either continue the chain, short-circuit with an error, or modify the security context.

HTTP Request │ ▼ ┌──────────────────────────────────────────────┐ │ SecurityFilterChain │ │ ┌──────────────────────────────────────┐ │ │ │ DisableEncodeUrlFilter │ │ │ │ WebAsyncManagerIntegrationFilter │ │ │ │ SecurityContextHolderFilter │ │ ← restore SecurityContext per request │ │ HeaderWriterFilter │ │ │ │ CorsFilter │ │ │ │ CsrfFilter │ │ │ │ AbstractPreAuthenticatedProcessingFilter│ │ ← YOUR custom filter (PoC) │ │ UsernamePasswordAuthenticationFilter │ │ │ │ ExceptionTranslationFilter │ │ ← catches auth exceptions │ │ AuthorizationFilter │ │ ← checks permissions │ └──────────────────────────────────────┘ │ └──────────────────────────────────────────────┘ │ ▼ DispatcherServlet → Controller

The key insight: Spring Security is filter-based. You integrate by inserting the right filter at the right position in the chain. For pre-authentication, your custom filter runs before UsernamePasswordAuthenticationFilter.

2 SecurityFilterChain

A SecurityFilterChain is a bean that maps URL patterns to a chain of security filters. You configure it using the HttpSecurity DSL inside a @Configuration class. Spring Security 6+ (and 7.x) removed WebSecurityConfigurerAdapter — you declare @Bean methods returning SecurityFilterChain instead.

@Configuration
@EnableWebSecurity
@EnableMethodSecurity  // enables @PreAuthorize on methods
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(AbstractHttpConfigurer::disable)          // REST APIs: disable CSRF
            .sessionManagement(s -> s
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/actuator/health").permitAll()
                .requestMatchers("/v1/products/**").authenticated()
                .anyRequest().authenticated()
            )
            .exceptionHandling(ex -> ex
                .authenticationEntryPoint(new Http403ForbiddenEntryPoint()))
            .build();
    }
}
Key API changes from Spring Security 5.x → 6.x/7.x
  • authorizeRequests() → authorizeHttpRequests()
  • antMatchers() → requestMatchers()
  • WebSecurityConfigurerAdapter removed — use @Bean SecurityFilterChain
  • csrf().disable() → csrf(AbstractHttpConfigurer::disable)
  • @EnableGlobalMethodSecurity(prePostEnabled=true) → @EnableMethodSecurity

Multiple SecurityFilterChains

You can have multiple SecurityFilterChain beans with different URL patterns. Order them with @Order — the first chain whose pattern matches wins. Useful for having one chain for actuator endpoints (no auth) and another for API endpoints (pre-auth required).

3 AuthenticationManager

The AuthenticationManager is the orchestrator of authentication. It takes an unauthenticated Authentication token and returns an authenticated one (or throws AuthenticationException).

PreAuthenticatedAuthenticationToken (unauthenticated) │ ▼ ProviderManager (standard AuthenticationManager impl) │ ┌───────────┴───────────────┐ ▼ ▼ PreAuthenticatedAuthenticationProvider DaoAuthenticationProvider (handles PreAuth tokens) (handles UsernamePasswordToken) │ ▼ PreAuthenticatedAuthenticationToken (authenticated ✓)

ProviderManager tries each registered AuthenticationProvider in order. It skips providers that return false from supports(Class). For pre-auth, you only need one provider.

@Bean
public AuthenticationManager authenticationManager() {
    return new ProviderManager(preAuthProvider());
}

4 AuthenticationProvider

Each AuthenticationProvider handles a specific type of Authentication token. The interface has two methods:

public interface AuthenticationProvider {
    // Perform authentication. Return authenticated token or throw AuthenticationException.
    Authentication authenticate(Authentication authentication) throws AuthenticationException;

    // Declares which token type this provider can handle.
    boolean supports(Class<?> authentication);
}

For pre-authentication, PreAuthenticatedAuthenticationProvider is the built-in provider. It:

  1. Checks that it supports PreAuthenticatedAuthenticationToken
  2. Calls AuthenticationUserDetailsService.loadUserDetails(token) to get UserDetails
  3. Returns a fully authenticated PreAuthenticatedAuthenticationToken
@Bean
public PreAuthenticatedAuthenticationProvider preAuthProvider() {
    PreAuthenticatedAuthenticationProvider provider =
        new PreAuthenticatedAuthenticationProvider();
    provider.setPreAuthenticatedUserDetailsService(userDetailsService);
    return provider;
}

5 UserDetailsService & UserDetails

UserDetails is the contract Spring Security uses to represent a user internally. It carries the username, password (not needed in pre-auth), whether the account is enabled/locked, and the list of GrantedAuthority objects (roles/permissions).

public interface UserDetails extends Serializable {
    String getUsername();
    String getPassword();  // empty string for pre-auth
    Collection<? extends GrantedAuthority> getAuthorities();
    boolean isAccountNonExpired();
    boolean isAccountNonLocked();
    boolean isCredentialsNonExpired();
    boolean isEnabled();
}

Spring provides a builder: User.withUsername("alice").password("").authorities("ROLE_USER").build()

UserDetailsServiceString usernameDaoAuthenticationProvider, UserDetailsByNameServiceWrapper

For pre-auth, use AuthenticationUserDetailsService<PreAuthenticatedAuthenticationToken> — it gives you access to the full token including the details object where you can store header-derived data.

6 GrantedAuthority

A GrantedAuthority is simply a string permission/role. The convention for roles is the ROLE_ prefix — hasRole("ADMIN") checks for ROLE_ADMIN in the authority list.

// Creating authorities
GrantedAuthority role = new SimpleGrantedAuthority("ROLE_ADMIN");

// In controller/service — access current auth
Authentication auth = SecurityContextHolder.getContext().getAuthentication();
boolean isAdmin = auth.getAuthorities().stream()
    .anyMatch(a -> a.getAuthority().equals("ROLE_ADMIN"));

// Via @PreAuthorize (requires @EnableMethodSecurity)
@PreAuthorize("hasRole('ADMIN')")
public void adminOnlyMethod() { ... }

// In @PreAuthorize — access current principal
@PreAuthorize("authentication.name == #username")
public Data getOwnData(String username) { ... }

7 SecurityContext & SecurityContextHolder

The SecurityContext holds the current Authentication object for the duration of a request. SecurityContextHolder is the static accessor — by default it's thread-local, so each request thread has its own context.

// Reading current auth in any bean
Authentication auth = SecurityContextHolder.getContext().getAuthentication();
String username = auth.getName();  // returns principal.toString()

// In controller — inject via @AuthenticationPrincipal
@GetMapping("/me")
ResponseEntity<String> me(@AuthenticationPrincipal UserDetails user) {
    return ResponseEntity.ok(user.getUsername());
}

// Setting context manually (in custom filters)
SecurityContext ctx = SecurityContextHolder.createEmptyContext();
ctx.setAuthentication(authentication);
SecurityContextHolder.setContext(ctx);
// → Always clear after setting in tests or manual flows
SecurityContextHolder.clearContext();
Async / Reactive Warning
Thread-local context does not propagate to @Async methods or reactive streams automatically. For async: wrap with DelegatingSecurityContextRunnable. For WebFlux (not applicable here): use ReactorContextHolder.

8 Pre-Authentication: Overview & When to Use

Pre-authentication is the pattern where an external system has already authenticated the user before the request reaches your application. Your app's job is only to:

  1. Extract the pre-authenticated identity from the request
  2. Load application-specific roles/permissions for that identity
  3. Store the authenticated principal in the SecurityContext

Common scenarios where this applies:

ScenarioIdentity Delivery
Veridium + WAF/reverse proxyCustom HTTP headers set by proxy after biometric auth
Siteminder / CA Access ManagerSM_USER header
Kerberos / SPNEGO (IWA)Kerberos ticket negotiated at proxy, username injected
Client certificate (mTLS)Certificate DN extracted by servlet container
JWT from external IdP (OAuth2 RS)Authorization: Bearer <jwt> header
Your scenario: IWA with Veridium
The product module has samAccountName as its user identifier — this is an Active Directory attribute. Veridium performs biometric authentication, verifies the AD identity, and (via WAF/proxy) injects the samAccountName into a trusted HTTP header. Spring Security's pre-auth filter picks this up.

9 AbstractPreAuthenticatedProcessingFilter

This is the base class you extend to create a pre-auth filter. It lives in org.springframework.security.web.authentication.preauth.

How it works

1. Request arrives. Filter checks: does SecurityContext already have a valid authentication?
If yes AND principal hasn't changed → skip. If checkForPrincipalChanges=true, re-auth if principal differs.
↓
2. Calls getPreAuthenticatedPrincipal(request) — your implementation
Return the username/identity extracted from the request. Return null if not present.
↓
3. If principal is null → no pre-auth present. If exceptionIfHeaderMissing=true throws; otherwise continues chain unauthenticated.
↓
4. Calls getPreAuthenticatedCredentials(request) — your implementation
Usually returns "N/A" — credentials not meaningful when external system authenticated.
↓
5. Creates unauthenticated PreAuthenticatedAuthenticationToken(principal, credentials)
↓
6. Calls buildDetails(request) and sets token.details — override to store extra data (e.g. roles from headers)
↓
7. Submits token to AuthenticationManager.authenticate(token)
↓
8. On success: sets authenticated token in SecurityContextHolder. Continues filter chain.
On failure: calls unsuccessfulAuthentication() which clears context and calls entry point.

The two abstract methods

public class VeridiumPreAuthFilter extends AbstractPreAuthenticatedProcessingFilter {

    @Override
    protected Object getPreAuthenticatedPrincipal(HttpServletRequest request) {
        // Extract identity from request. Return null → no pre-auth → continue unauthenticated.
        return request.getHeader("X-Authenticated-User");
    }

    @Override
    protected Object getPreAuthenticatedCredentials(HttpServletRequest request) {
        // Not meaningful for pre-auth — external system already authenticated
        return "N/A";
    }
}

Built-in subclass: RequestHeaderAuthenticationFilter

Spring provides RequestHeaderAuthenticationFilter as a ready-to-use implementation that reads a configurable header. Configured by setPrincipalRequestHeader(String). Use this when you have no need for custom logic beyond reading a single header.

RequestHeaderAuthenticationFilter filter = new RequestHeaderAuthenticationFilter();
filter.setPrincipalRequestHeader("X-Authenticated-User");
filter.setCredentialsRequestHeader("X-Auth-Token");  // optional
filter.setExceptionIfHeaderMissing(false);  // don't throw if header absent
filter.setAuthenticationManager(authenticationManager());

10 PreAuthenticatedAuthenticationProvider

Handles PreAuthenticatedAuthenticationTokens. It does very little — delegates loading of user details to AuthenticationUserDetailsService. The loaded UserDetails object's authorities become the authenticated user's permissions.

PreAuthenticatedAuthenticationProvider provider =
    new PreAuthenticatedAuthenticationProvider();

// Option A: your custom implementation
provider.setPreAuthenticatedUserDetailsService(myVeridiumUserDetailsService);

// Option B: wrap existing UserDetailsService (looks up by username only)
provider.setPreAuthenticatedUserDetailsService(
    new UserDetailsByNameServiceWrapper<>(myUserDetailsService));

// Option C: use authorities from the GrantedAuthoritiesContainer in token.details
provider.setPreAuthenticatedUserDetailsService(
    new PreAuthenticatedGrantedAuthoritiesUserDetailsService());

11 AuthenticationUserDetailsService

The interface your service implements to load UserDetails for a pre-authenticated token:

public interface AuthenticationUserDetailsService<T extends Authentication> {
    UserDetails loadUserDetails(T token) throws UsernameNotFoundException;
}

// For pre-auth, T is PreAuthenticatedAuthenticationToken
@Service
class VeridiumUserDetailsService
    implements AuthenticationUserDetailsService<PreAuthenticatedAuthenticationToken> {

    @Override
    public UserDetails loadUserDetails(PreAuthenticatedAuthenticationToken token) {
        String samAccountName = (String) token.getPrincipal();

        // Load roles from DB or derive from token.getDetails()
        List<GrantedAuthority> authorities = resolveAuthorities(samAccountName, token);

        return User.withUsername(samAccountName)
            .password("")
            .authorities(authorities)
            .build();
    }
}

The advantage of AuthenticationUserDetailsService over plain UserDetailsService: you have access to the full token including its details object, which can carry roles extracted from HTTP headers.

12 Authorities from HTTP Headers (GrantedAuthoritiesContainer)

When the upstream proxy sets both the username and role headers, you can skip the database lookup entirely by carrying roles in the token's details object.

Pattern: Override buildDetails(request) in your filter to return a custom object implementing GrantedAuthoritiesContainer. Then use PreAuthenticatedGrantedAuthoritiesUserDetailsService as the provider's UDS — it reads authorities directly from the container.

// 1. Implement GrantedAuthoritiesContainer on your details object
class VeridiumAuthDetails extends WebAuthenticationDetails
        implements GrantedAuthoritiesContainer {

    private final List<GrantedAuthority> authorities;

    VeridiumAuthDetails(HttpServletRequest request) {
        super(request);
        String rolesHeader = request.getHeader("X-User-Roles");
        this.authorities = rolesHeader != null
            ? Arrays.stream(rolesHeader.split(","))
                .map(String::trim)
                .filter(s -> !s.isBlank())
                .map(r -> new SimpleGrantedAuthority("ROLE_" + r.toUpperCase()))
                .<GrantedAuthority>toList()
            : List.of(new SimpleGrantedAuthority("ROLE_USER"));
    }

    @Override
    public Collection<? extends GrantedAuthority> getGrantedAuthorities() {
        return authorities;
    }
}

// 2. Override buildDetails in your filter
@Override
protected Object buildDetails(HttpServletRequest request) {
    return new VeridiumAuthDetails(request);
}

// 3. Use PreAuthenticatedGrantedAuthoritiesUserDetailsService — reads from the container
provider.setPreAuthenticatedUserDetailsService(
    new PreAuthenticatedGrantedAuthoritiesUserDetailsService());

14 Veridium — What Protocols It Supports

ProtocolVersionHow Identity is Delivered
SAML 2.0 v3.6+ SAML Assertion POSTed to your ACS endpoint; contains Subject (username) + attribute statements (AD groups, roles)
OpenID Connect / OAuth2 v3.7+ JWT Bearer Token (at+jwt type); claims: sub, aud, scope, exp, custom group claims
REST API identityToken v3.6+ Server-signed token in GetSessionStatus response; subject + authenticator info + timestamp
SSO Header (proxy) v3.7+ Custom HTTP headers set by WAF/proxy after authentication: username + group headers
mTLS + WAF v3.7.2+ X-SSL-Client-DN header with certificate DN + proxy secret header
SAML and OIDC are mutually exclusive in Veridium
A single Veridium instance supports either SAML or OIDC — not both simultaneously. Choose based on your environment's capabilities. SAML is more common in enterprise; OIDC is preferred for microservices.

15 Pattern A: SAML 2.0 (Browser SSO)

Use when Veridium is your enterprise SSO IdP for browser-based web apps. Users see a Veridium login page, authenticate biometrically, then get redirected back to your app with a SAML assertion.

// pom.xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.security</groupId>

spring:
  security:
    saml2:
      relyingparty:
        registration:
          veridium:
            asserting-party:
              metadata-uri: https://<veridium-host>/idp/shibboleth
            entity-id: https://<your-app>/saml2/service-provider-metadata/veridium
            acs:
              location: https://<your-app>/login/saml2/sso/veridium
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    return http
        .authorizeHttpRequests(auth -> auth.anyRequest().authenticated())
        .saml2Login(Customizer::withDefaults)
        .saml2Logout(Customizer::withDefaults)
        .build();
}

// Access attributes from SAML assertion in controller
@GetMapping("/me")
String me(@AuthenticationPrincipal Saml2AuthenticatedPrincipal principal) {
    String username = principal.getName();
    List<String> groups = principal.getAttribute("groups");
    return username;
}

16 Pattern B: OIDC JWT (API Resource Server)

Use when Veridium issues JWTs and your Spring Boot app is a stateless REST API. The client presents a JWT Bearer token with each request.

Veridium JWT quirk: at+jwt token type
Veridium OIDC issues tokens with type at+jwt. Spring Security's default JWT decoder rejects non-standard types. You must customize the JWS type verifier. 
@Bean
public JwtDecoder veridiumJwtDecoder() {
    NimbusJwtDecoder decoder = NimbusJwtDecoder
        .withJwkSetUri("https://<veridium-host>/idp/profile/oidc/keyset")
        .jwtProcessorCustomizer(processor -> {
            processor.setJWSTypeVerifier(
                new DefaultJOSEObjectTypeVerifier<>(
                    new JOSEObjectType("at+jwt"),  // Veridium's type
                    JOSEObjectType.JWT,
                    null
                )
            );
        })
        .build();
    return decoder;
}

@Bean
public JwtAuthenticationConverter jwtConverter() {
    JwtGrantedAuthoritiesConverter grantedAuthConverter = new JwtGrantedAuthoritiesConverter();
    grantedAuthConverter.setAuthoritiesClaimName("groups");  // your Veridium claim name
    grantedAuthConverter.setAuthorityPrefix("ROLE_");
    JwtAuthenticationConverter converter = new JwtAuthenticationConverter();
    converter.setJwtGrantedAuthoritiesConverter(grantedAuthConverter);
    return converter;
}

21 Step 2 — VeridiumPreAuthFilter

This filter reads the pre-authenticated identity from the X-Authenticated-User header.

package com.iwa.products.security;

import jakarta.servlet.http.HttpServletRequest;
import org.springframework.security.web.authentication.preauth.AbstractPreAuthenticatedProcessingFilter;

class VeridiumPreAuthFilter extends AbstractPreAuthenticatedProcessingFilter {

    static final String PRINCIPAL_HEADER = "X-Authenticated-User";

    @Override
    protected Object getPreAuthenticatedPrincipal(HttpServletRequest request) {
        // Returning null means "no pre-auth found" — chain continues unauthenticated
        return request.getHeader(PRINCIPAL_HEADER);
    }

    @Override
    protected Object getPreAuthenticatedCredentials(HttpServletRequest request) {
        return "N/A";
    }

    @Override
    protected Object buildDetails(HttpServletRequest request) {
        // Carries roles extracted from headers — consumed by VeridiumUserDetailsService
        return new VeridiumAuthDetails(request);
    }
}

22 Step 3 — VeridiumAuthDetails

Carries the roles from the X-User-Roles header. Implements GrantedAuthoritiesContainer so the provider can extract them without a DB call.

package com.iwa.products.security;

import jakarta.servlet.http.HttpServletRequest;
import org.springframework.security.core.GrantedAuthority;
import org.springframework.security.core.authority.SimpleGrantedAuthority;
import org.springframework.security.web.authentication.WebAuthenticationDetails;
import org.springframework.security.web.authentication.preauth.PreAuthenticatedGrantedAuthoritiesWebAuthenticationDetails;

import java.util.Arrays;
import java.util.Collection;
import java.util.List;

class VeridiumAuthDetails extends WebAuthenticationDetails
        implements org.springframework.security.web.authentication.preauth.PreAuthenticatedGrantedAuthoritiesWebAuthenticationDetails {

    static final String ROLES_HEADER = "X-User-Roles";

    private final List<GrantedAuthority> authorities;

    VeridiumAuthDetails(HttpServletRequest request) {
        super(request);
        String rolesHeader = request.getHeader(ROLES_HEADER);
        authorities = rolesHeader != null && !rolesHeader.isBlank()
            ? Arrays.stream(rolesHeader.split(","))
                .map(String::trim)
                .filter(s -> !s.isBlank())
                .map(r -> (GrantedAuthority) new SimpleGrantedAuthority("ROLE_" + r.toUpperCase()))
                .toList()
            : List.of(new SimpleGrantedAuthority("ROLE_USER"));
    }

    @Override
    public Collection<? extends GrantedAuthority> getGrantedAuthorities() {
        return authorities;
    }
}

23 Step 4 — VeridiumUserDetailsService

Loads UserDetails for a pre-authenticated token. Reads authorities from the VeridiumAuthDetails object attached to the token. In a real system, you might also cross-reference a local DB for application-specific roles.

package com.iwa.products.security;

import org.springframework.security.core.userdetails.AuthenticationUserDetailsService;
import org.springframework.security.core.userdetails.User;
import org.springframework.security.core.userdetails.UserDetails;
import org.springframework.security.core.userdetails.UsernameNotFoundException;
import org.springframework.security.web.authentication.preauth.PreAuthenticatedAuthenticationToken;
import org.springframework.stereotype.Service;

@Service
class VeridiumUserDetailsService
        implements AuthenticationUserDetailsService<PreAuthenticatedAuthenticationToken> {

    @Override
    public UserDetails loadUserDetails(PreAuthenticatedAuthenticationToken token)
            throws UsernameNotFoundException {
        String samAccountName = (String) token.getPrincipal();

        if (samAccountName == null || samAccountName.isBlank()) {
            throw new UsernameNotFoundException("No samAccountName in pre-auth token");
        }

        var authorities = token.getDetails() instanceof VeridiumAuthDetails details
            ? details.getGrantedAuthorities()
            : java.util.List.of();

        return User.withUsername(samAccountName)
            .password("")
            .authorities(authorities)
            .build();
    }
}

24 Step 5 — SecurityConfiguration

Wires everything together: the filter, provider, and SecurityFilterChain.

package com.iwa.products.configuration;

import com.iwa.products.security.VeridiumPreAuthFilter;
import com.iwa.products.security.VeridiumUserDetailsService;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.authentication.AuthenticationManager;
import org.springframework.security.authentication.ProviderManager;
import org.springframework.security.config.annotation.method.configuration.EnableMethodSecurity;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.config.annotation.web.configurers.AbstractHttpConfigurer;
import org.springframework.security.config.http.SessionCreationPolicy;
import org.springframework.security.web.SecurityFilterChain;
import org.springframework.security.web.authentication.UsernamePasswordAuthenticationFilter;
import org.springframework.security.web.authentication.preauth.PreAuthenticatedAuthenticationProvider;
import org.springframework.security.web.authentication.Http403ForbiddenEntryPoint;

@Configuration
@EnableWebSecurity
@EnableMethodSecurity
class SecurityConfiguration {

    private final VeridiumUserDetailsService userDetailsService;

    SecurityConfiguration(VeridiumUserDetailsService userDetailsService) {
        this.userDetailsService = userDetailsService;
    }

    @Bean
    SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(AbstractHttpConfigurer::disable)
            .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .addFilterBefore(veridiumPreAuthFilter(), UsernamePasswordAuthenticationFilter.class)
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/actuator/health").permitAll()
                .anyRequest().authenticated()
            )
            .exceptionHandling(ex -> ex
                .authenticationEntryPoint(new Http403ForbiddenEntryPoint()))
            .build();
    }

    @Bean
    VeridiumPreAuthFilter veridiumPreAuthFilter() {
        VeridiumPreAuthFilter filter = new VeridiumPreAuthFilter();
        filter.setAuthenticationManager(authenticationManager());
        filter.setExceptionIfHeaderMissing(false);  // continue unauthenticated if header absent
        filter.setCheckForPrincipalChanges(true);   // re-auth if principal header changes
        return filter;
    }

    @Bean
    AuthenticationManager authenticationManager() {
        return new ProviderManager(preAuthProvider());
    }

    @Bean
    PreAuthenticatedAuthenticationProvider preAuthProvider() {
        PreAuthenticatedAuthenticationProvider provider =
            new PreAuthenticatedAuthenticationProvider();
        provider.setPreAuthenticatedUserDetailsService(userDetailsService);
        return provider;
    }
}

27 Security Hardening for Header Pre-Auth

Critical: Spring Security does NOT validate header authenticity
It trusts whatever value is in the header. Anyone who can reach your service and spoof the headers can authenticate as any user. You MUST ensure the service is only reachable from trusted sources (the proxy).

Hardening checklist

  • Network isolation: Product service (port 8081) must not be reachable directly from the internet. Bind to internal interface only, or use firewall rules.
  • Proxy secret header: Add a shared secret header (X-Proxy-Secret) that the WAF sets. Validate it in a filter that runs before VeridiumPreAuthFilter. Reject if missing or wrong.
  • Header stripping: The WAF/proxy must strip any client-provided X-Authenticated-User and X-User-Roles headers before adding its own. Configure this in NGINX/F5/Netscaler.
  • mTLS between proxy and service: Require client certificate from the proxy. Validate it at the Spring Boot layer using the servlet container's SSL config.
  • Immutable headers validation filter:
class ProxySecretValidationFilter extends OncePerRequestFilter {

    private final String expectedSecret;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
            HttpServletResponse response, FilterChain chain)
            throws ServletException, IOException {
        String secret = request.getHeader("X-Proxy-Secret");
        if (!expectedSecret.equals(secret)) {
            response.sendError(HttpServletResponse.SC_FORBIDDEN, "Missing proxy secret");
            return;
        }
        chain.doFilter(request, response);
    }
}

// In SecurityConfiguration — add BEFORE veridiumPreAuthFilter
.addFilterBefore(proxySecretFilter, VeridiumPreAuthFilter.class)
