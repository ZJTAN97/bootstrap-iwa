MinIO Retry Design

Why RetryTemplate beats @Retryable for MinioService

Verdict: Keep the programmatic RetryTemplate. The @Retryable annotation looks simpler on the surface, but it fights four properties this service depends on. Switching would make the code more complex, not less, and would quietly change retry behaviour.

The current design

Every public method funnels through one private execute(...) helper. That helper does two jobs: it runs the call (optionally wrapped in retry) and it translates the MinIO SDK's checked exceptions into the project's MinioStorageException hierarchy. 

private <T> T execute(String bucket, String objectPath, String operation,
                     boolean retryable, MinioCall<T> call) {
    try {
        return retryable ? retryTemplate.execute(ctx -> call.call()) : call.call();
    } catch (ErrorResponseException e) {
        throw translate(bucket, objectPath, operation, e);   // SDK -> domain exception
    } catch (Exception e) {
        throw new MinioStorageException(...);
    }
}

The key detail: the RetryTemplate wraps only the raw SDK call (ctx -> call.call()). Exception translation happens outside the retry, in the surrounding catch blocks. The retry classifier therefore sees the original IOException / ServerException — never the translated domain exception.

Why @Retryable is the wrong fit

1. Per-call retry toggle

execute(...) takes a boolean retryable. Idempotent reads (getObject, statObject, listObjects) retry; stream uploads and non-idempotent writes (putObject) pass false and never retry — retrying a half-consumed stream would corrupt the upload.

@Retryable is fixed per method. To get the same behaviour you would have to split retrying and non-retrying operations into separate annotated methods, or duplicate logic. The boolean flag keeps it to a single code path.

2. The classifier must see the RAW SDK exception

With RetryTemplate, retry decisions are made on IOException / ServerException — before translation. A deterministic NoSuchKey error becomes ObjectNotFoundException and is not retried (it would never succeed).

@Retryable works through a Spring AOP proxy that wraps the entire public method. Translation runs inside the advised method, so by the time the exception reaches the retry logic it is already a MinioStorageException. You would be forced to classify retry on the translated type and lose the clean raw-exception distinction.

RetryTemplate (current)
call() → IOException → retry sees IOException → RETRY → … → translate → MinioStorageException

@Retryable
[proxy[ call() → IOException → translate → MinioStorageException ]] → retry sees MinioStorageException (too late, info lost)

3. Self-invocation proxy trap

Spring AOP only applies @Retryable when a method is called from outside the bean, through the proxy. Internal calls — e.g. presignedGetUrl delegating to the private presignedUrl helper — bypass the proxy and would silently get no retry. The programmatic template has no such trap: retry is wherever you write retryTemplate.execute(...).

4. Tests stay fast and context-free

MinioServiceRetryTests constructs a no-backoff RetryTemplate and injects it straight into the service constructor with a mocked MinioClient. No Spring context, no sleeping, fully deterministic. 

RetryTemplate retryTemplate = RetryTemplate.builder()
        .maxAttempts(3)
        .noBackoff()
        .retryOn(ServerException.class)
        .retryOn(IOException.class)
        .traversingCauses()
        .build();
minioService = new MinioService(minioClient, retryTemplate);

@Retryable only activates inside a Spring context with @EnableRetry, so the equivalent test needs @SpringBootTest (or manual proxy wiring) — slower and heavier just to assert "it retried three times".

When @Retryable would win: a uniform retry policy across all methods, retrying on the translated exception is acceptable, no per-call toggle is needed, and all calls arrive through the proxy. None of those hold for MinioService.

Bottom line KEEP RetryTemplate

The annotation trades a tiny amount of boilerplate for loss of control over what gets retried, which exception drives the decision, and where retry actually applies — plus heavier tests. For a thin SDK wrapper whose whole point is precise exception handling, the explicit RetryTemplate is both simpler in practice and safer.
